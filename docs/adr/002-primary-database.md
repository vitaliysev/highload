# ADR-002: Выбор основной базы данных

## Title
Выбор основной СУБД для транзакционного хранилища: PostgreSQL

## Status
Accepted

## Context

Системе требуется основное хранилище для следующих сущностей и паттернов доступа:

**Объявления (`listings`):**
- Запись одной строки при создании (~0.7 RPS средний, ~3.5 RPS пик).
- Точечное чтение по PK при отдаче карточки (~1 040 RPS, но большинство закрывается Redis-кэшем).
- UPDATE по PK при смене статуса (модерация, архивация).
- FTS-запросы при деградации Elasticsearch (fallback).

**Пользователи (`users`):** точечное чтение по email/phone при аутентификации.

**Платежи (`payments`):** INSERT при инициации, UPDATE при получении webhook, критически важна идемпотентность через `unique constraint` на `external_payment_id`. ACID обязателен.

**Продвижения (`promotions`):** INSERT при активации, SELECT с фильтром по `active` и `expires_at`.

**Фото (`listing_photos`):** INSERT при подтверждении загрузки, SELECT по `listing_id` при отдаче карточки.

**Объёмы:**
- Объявления: ~10 млн строк за 3 года, ~40 GB с индексами.
- Фото-метаданные: ~50 млн строк, ~25 GB.
- Итого PostgreSQL: ~130 GB за 3 года — легко размещается на одном узле с репликой.

**Требования к надёжности:** durability 99.999999999% для объявлений и пользователей; платежи не теряются ни при каких обстоятельствах (НФТ-004).

## Decision

Выбрана **PostgreSQL 16** в качестве единственной основной транзакционной СУБД.

Конкретные особенности использования:
- Синхронная репликация на одну standby-реплику (Yandex Cloud Managed PostgreSQL) — гарантирует durability платежей.
- `unique constraint` на `payments.external_payment_id` — ключевой механизм идемпотентной обработки webhook.
- `GIN`-индекс на `tsvector`-колонке в `listings` — FTS fallback при недоступности Elasticsearch.
- Read-replica для Listing Service (чтение карточек при cache miss) — снижает нагрузку на primary.

## Alternatives Considered

### Вариант 1: MongoDB

**Преимущества**: гибкая схема (удобно для разных категорий объявлений с разными атрибутами), горизонтальное шардирование из коробки.

**Недостатки**:
- До MongoDB 4.0 нет ACID для мультидокументных транзакций; с 4.0+ есть, но с overhead и ограничениями.
- Наша модель данных реляционная по природе: объявление → фото → продвижение → платёж. JOIN в MongoDB — `$lookup`, который медленнее и сложнее нативных SQL JOIN.
- `unique constraint` на уровне СУБД в MongoDB менее надёжен в шардированном кластере.
- Для объёма ~130 GB реляционная модель не является узким местом — гибкость схемы не нужна.

**Вывод**: не даёт преимуществ под наш паттерн, но добавляет сложность транзакций.

### Вариант 2: Cassandra (Apache Cassandra / ScyllaDB)

**Преимущества**: отличная горизонтальная масштабируемость, высокая пропускная способность записи, отказоустойчивость без single point of failure.

**Недостатки**:
- Cassandra — wide-column store, оптимизирована для append-only write-heavy нагрузки. У нас writes ~3.5 RPS пик — это не проблема для одного узла PostgreSQL.
- Нет поддержки ACID-транзакций в классическом смысле; lightweight transactions (LWT) — дорогие и ненадёжны при сетевых разделениях.
- Нет поддержки JOIN — вся денормализация вручную.
- Idempotency через `IF NOT EXISTS` менее удобна, чем `unique constraint` + `FOR UPDATE` в PostgreSQL.
- Оперировать Cassandra сложнее: compaction, tombstones, tuning consistency levels — требует специализированной экспертизы.

**Вывод**: избыточна для write-нагрузки ~3.5 RPS; ACID-гарантии для платежей реализовать сложнее.

### Вариант 3 (выбранный): PostgreSQL

Зрелая, хорошо понятая СУБД с ACID из коробки, нативными JOIN, `unique constraint`, `SELECT FOR UPDATE`, `tsvector` для FTS fallback. Managed PostgreSQL в Yandex Cloud закрывает вопрос надёжности без операционного overhead.

## Consequences

**Положительные:**
- ACID гарантирует корректность платёжных операций и идемпотентность webhook.
- `tsvector` + GIN-индекс обеспечивает FTS fallback при недоступности Elasticsearch.
- Знакомый стек для большинства Go-разработчиков — снижает время до продакшна.
- Managed PostgreSQL (Yandex Cloud) закрывает вопрос репликации, резервных копий и мониторинга.
- Объём ~130 GB за 3 года — без шардирования, на одном primary + read-replica.

**Отрицательные:**
- При росте выше 30 млн MAU и соответствующем росте write-нагрузки может потребоваться шардирование или переход к распределённой СУБД (CockroachDB, YDB).
- PostgreSQL не является идеальным search-движком — Elasticsearch остаётся обязательным для production-качества поиска. PostgreSQL — только fallback.
- Вертикальное масштабирование primary ограничено — необходимо отслеживать размер соединений и использовать connection pooler (PgBouncer).
