# ADR-001: Выбор архитектурного стиля

## Title
Выбор архитектурного стиля: Service-Based Architecture с элементами Event-Driven

## Status
Accepted

## Context

Система представляет собой маркетплейс объявлений с 10–30 млн MAU. Ключевые характеристики нагрузки:

- **Read-heavy**: поиск — 700 RPS средний / 3 500 RPS пик; просмотр карточек — 1 040 RPS / 5 000 RPS пик.
- **Write-light**: создание объявлений — ~0.7 RPS; оплата — ~0.02 RPS.
- **Разные SLA**: поиск — 99.95%, оплата — 99.99% (данные), модерация — 99.9%.
- **Разные latency-требования**: поиск p99 ≤ 300 ms, карточка p99 ≤ 250 ms, модерация — асинхронно ≤ 30 сек.
- **Ограниченный бюджет**: предпочтение open-source, облачная инфраструктура Yandex Cloud.
- **Асинхронные процессы**: модерация контента (ML API, ручная проверка) и обработка webhook от платёжного шлюза не должны блокировать основные пользовательские запросы.

Требуется возможность масштабировать read-путь (поиск, карточки) независимо от write-пути (создание объявлений, оплата) — в монолите это невозможно без специальных ухищрений.

## Decision

Выбрана **Service-Based Architecture (SBA)** с элементами **Event-Driven Architecture (EDA)**.

Система разделена на три прикладных сервиса с чёткими границами ответственности:
- **Listing Service** — жизненный цикл объявлений, фото, личный кабинет.
- **Search Service** — поиск и фильтрация.
- **Payment Service** — платежи и продвижение.

Event-Driven элементы применяются точечно:
- Модерация: Listing Service публикует задачу в RabbitMQ, Moderation Worker (встроен в Listing Service) консьюмит асинхронно.
- Уведомления: события публикуются в очередь `notifications`, консьюмер доставляет через SMS/Push провайдер.
- Активация продвижения: Payment Service публикует событие `promotion-activated`, Listing Service обновляет запись.

## Alternatives Considered

### Вариант 1: Layered Monolith (Слоистый монолит)

**Преимущества**: минимальная операционная сложность, простой деплой, нет сетевых вызовов между компонентами.

**Недостатки**:
- Невозможно масштабировать Search отдельно от Listing при пике 5 000 RPS на карточки и 3 500 RPS на поиск — придётся масштабировать всё приложение целиком, что дорого.
- Асинхронная модерация потребует внутренних воркеров — усложняет монолит.
- Один деплой на всё — любое изменение в Payment-логике требует деплоя всего приложения.

**Вывод**: не подходит из-за ограничений по независимому масштабированию при нашем RPS.

### Вариант 2: Microservices Architecture (MSA)

**Преимущества**: максимальная независимость компонентов, гранулярное масштабирование.

**Недостатки**:
- Требует service mesh (Istio/Linkerd), оркестрацию (Kubernetes), централизованный distributed tracing и мониторинг — значительные операционные затраты.
- При ограниченном бюджете невозможно обеспечить команду для эксплуатации инфраструктуры такого уровня.
- Десятки мелких сервисов при нашем RPS создают overhead на сетевые вызовы и latency.
- Распределённые транзакции (saga, 2PC) усложняют гарантии консистентности платежей.

**Вывод**: избыточна для данного этапа и бюджета. При росте до 50+ млн MAU можно эволюционировать к MSA.

### Вариант 3 (выбранный): Service-Based Architecture + Event-Driven элементы

Три крупных сервиса с общей PostgreSQL, брокером RabbitMQ для async и Elasticsearch для поиска. Каждый сервис масштабируется горизонтально независимо. Операционная сложность умеренная — нет service mesh, нет распределённых транзакций.

## Consequences

**Положительные:**
- Search Service и Listing Service масштабируются независимо — выполняется НФТ-005.
- Асинхронная модерация не блокирует пользователя при создании объявления.
- Умеренная операционная сложность — доступна небольшой команде при ограниченном бюджете.
- Возможность эволюции: при необходимости отдельные модули вырезаются в самостоятельные микросервисы.

**Отрицательные:**
- Три сервиса вместо одного — больше точек отказа, нужен мониторинг каждого.
- Отсутствие service mesh означает ручное управление retry, таймаутами и circuit breaker на уровне кода.
- Shared PostgreSQL — потенциальный bottleneck при масштабировании выше 30 млн MAU. Решение: read-replica для Listing Service и Payment Service.
