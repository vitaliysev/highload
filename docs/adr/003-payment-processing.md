# ADR-003: Обработка платежей за продвижение объявлений

## Title
Стратегия обработки платежей: асинхронная через webhook с идемпотентной обработкой

## Status
Accepted

## Context

Система поддерживает платное продвижение объявлений — пользователь оплачивает повышение в поисковой выдаче через СПБ или банковскую карту.

**Требования:**
- SLA доступности: 99.99% — потеря платежей недопустима (НФТ-002).
- Durability транзакций: 100% — все транзакции логируются (НФТ-004).
- Latency инициации платежа: p99 ≤ 1 сек (НФТ-001).
- Объём: ~0.02 RPS средний, ~0.1 RPS пик — крайне низкая нагрузка.

**Ключевая проблема**: внешний платёжный шлюз (СПБ/эквайринг) является асинхронным по природе — он не возвращает финальный результат платежа синхронно. Пользователь переходит на страницу шлюза, оплачивает там, после чего шлюз уведомляет нашу систему через **webhook**. При этом:
- Шлюз может прислать webhook более одного раза (повторные попытки при таймаутах).
- Шлюз может прислать webhook с задержкой до нескольких минут.
- Система должна корректно обработать дублирующие webhook без двойного списания или двойной активации продвижения.

**Ограничение бюджета**: решение не должно требовать экзотической инфраструктуры (распределённые СУБД, Kafka, saga-оркестратор).

## Decision

Выбрана **асинхронная обработка через webhook с идемпотентной обработкой на уровне PostgreSQL**.

**Схема работы:**

1. **Инициация** (sync): Payment Service создаёт запись `payments` со статусом `pending` и уникальным `payment_id`. Запрашивает у шлюза `redirect_url`. Возвращает пользователю URL для редиректа.

2. **Оплата**: пользователь оплачивает на стороне шлюза (вне системы).

3. **Webhook** (async): шлюз отправляет `POST /api/v1/payments/webhook` с `gateway_payment_id` и статусом. Payment Service обрабатывает идемпотентно:
   - `SELECT ... FOR UPDATE WHERE external_payment_id = ?` — блокировка строки.
   - Если статус уже `paid` → ответить `200 OK` без повторной обработки.
   - Если статус `pending` → `UPDATE status = 'paid'`, `INSERT promotions`, публикация события в RabbitMQ.

4. **Активация** (async): Listing Service консьюмит событие `promotion-activated`, обновляет статус объявления, инвалидирует Redis-кэш.

**Гарантии идемпотентности:**
- `UNIQUE CONSTRAINT` на `payments.external_payment_id` — база данных физически не допустит дублирующей записи.
- `SELECT FOR UPDATE` — сериализует параллельные webhook-запросы на уровне строки.
- `UPDATE ... WHERE status = 'pending'` — условный UPDATE предотвращает повторную активацию.

## Alternatives Considered

### Вариант 1: Полностью синхронная обработка

**Суть**: Payment Service синхронно вызывает API шлюза, ждёт финального результата, возвращает пользователю `200 OK` с итогом платежа в одном HTTP-запросе.

**Преимущества**: простая логика, не нужны webhook и асинхронность.

**Недостатки**:
- Большинство российских платёжных провайдеров (СПБ, эквайринг) **не предоставляют синхронный API** — пользователь должен пройти аутентификацию в банковском приложении или ввести данные карты на странице шлюза. Это невозможно сделать синхронно в рамках одного HTTP-запроса нашей системы.
- Если шлюз отвечает медленно — пользователь ждёт, нарушается SLO ≤ 1 сек на инициацию.
- При разрыве соединения между клиентом и нашим сервером — неизвестен статус платежа.

**Вывод**: технически неприменим для СПБ и большинства эквайринговых решений.

### Вариант 2: Polling статуса

**Суть**: клиент или сервер периодически опрашивает шлюз о статусе платежа (`GET /payments/{id}/status`).

**Преимущества**: не нужен публичный webhook-endpoint, проще настроить в изолированной сети.

**Недостатки**:
- Высокая задержка подтверждения (зависит от интервала polling). При интервале 5 сек — пользователь ждёт до 5 сек после оплаты.
- Лишняя нагрузка на API шлюза — платёжные провайдеры ограничивают частоту polling.
- Сложнее обеспечить надёжность: если polling-воркер упал — статус не обновится.
- При нашем объёме (~900 платежей/день) overhead на polling есть, но терпим — однако webhook всё равно предпочтительнее по latency.

**Вывод**: подходит как fallback (если webhook не пришёл за 5 минут — делаем polling), но не как основная стратегия.

### Вариант 3 (выбранный): Async через webhook + идемпотентность

Стандартная индустриальная практика для платёжных интеграций. Соответствует API всех крупных российских провайдеров (Юкасса, Тинькофф, СБП). Идемпотентность через `unique constraint` + `FOR UPDATE` реализована на уровне PostgreSQL без дополнительной инфраструктуры.

## Consequences

**Положительные:**
- Инициация платежа отвечает пользователю быстро (≤ 300 ms p50) — запись в БД + вызов шлюза.
- Webhook-обработка идемпотентна — дублирующие вызовы не приводят к ошибкам и не требуют ручного вмешательства.
- Durability 100%: запись `payments` создаётся до редиректа пользователя — даже при падении системы после этого момента запись в БД сохранена.
- Соответствует API реальных российских платёжных провайдеров.

**Отрицательные:**
- Система должна иметь публичный HTTPS-endpoint для получения webhook от шлюза — требует корректной настройки TLS и firewall rules.
- Пользователь не узнаёт о результате мгновенно — нужен механизм уведомления (push/SMS через очередь `notifications`).
- Необходим fallback polling: если webhook не пришёл за N минут — запускать фоновую проверку статуса. Это добавляет небольшую сложность, но критично для надёжности.
- При ошибке в обработчике webhook (например, недоступна PostgreSQL) — шлюз получит `5xx` и повторит webhook. Обработчик должен быть идемпотентным к этому сценарию — что и обеспечено выбранной схемой.
